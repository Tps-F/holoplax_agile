# UX AS-IS / TO-BE（たたき台）

## AS-IS（現状）
- オンボーディングは4ステップ:
  - 1) 目的 + ワークスペース名
  - 2) 目標 + ルーティン
  - 3) やるべきこと3件
  - 4) AI下準備プレビュー
- 完了後に `/backlog` へ遷移
- Plan/Execute/Review の流れを明示

## ペイン（初見ユーザー）
- 「何から始めれば良いか」が視覚的に分かりづらい
- 目標→分解→実行の流れが体験できない
- AIが“何をしてくれるのか”が伝わらない
- 低リスク/承認の考え方が曖昧

## TO-BE（方向性）
### ナビゲーションの方針
- **Plan First** を主導線にする。
- 初期遷移は Plan（/backlog）に寄せる。
- レビューは /review に集約する。

### オンボーディング体験
1) **目的とルールの提示**
   - アジャイルの簡易フロー（Plan/Execute/Review）
   - AIは“下準備”を行い、重要操作は承認が必要と明示
2) **最初の目標 + ルーティン**
   - 目標（EPIC）と日次/週次ルーティンを1〜2件作成
3) **フォーカスキュー作成**
   - 「やるべきこと（3件）」をAI補助で選定
4) **AI下準備のプレビュー**
   - メール文の下書き/実装方針案などを例示
   - 低リスクの基準は「保守的 → 徐々に引き上げる」と案内
   - 初期基準はスコアで判定する

## AI下準備の置き場（候補）
1) **タスク詳細のモーダル（採用）**
   - 「AIで下準備」ボタン → 具体アウトプットをモーダル表示
   - 例: メール文/チェックリスト/実装方針

## 追加で決めたいこと
- 低リスクの初期基準（タグ/スコア/作業種別）

## 現状との差分メモ
- フォーカスキューは実装済み
- ROUTINE は Sprint に混ぜる運用を許容（別枠でも可）
- スコア数値はユーザーに見せず、優先度ラベルで提示する
